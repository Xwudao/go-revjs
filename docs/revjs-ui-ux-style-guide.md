# UI / UX 通用风格指令文档

这份文档用于把当前项目已经调好的 UI / UX 风格，沉淀成一套以后可直接复用、也可直接喂给 AI 的执行规则。

目标不是“做一个看起来差不多的网站”，而是复用这套已经验证过的设计语言：技术感、工具感、紧凑、高密度、可扫描、低噪音。

## 1. 一句话定义风格

这是一个“专业工具工作台”风格的界面系统：

- 不是营销页，不走大段叙事和品牌秀场。
- 不是后台表格系统，不走僵硬的企业 admin 模板。
- 是一个偏工程师工作台的产品表面：紧凑、冷静、专业、可连续操作。

可以把它理解成：

- 入口页像“技术工具集合入口”。
- 工具页像“可直接开工的工作台”。
- 视觉强调层级、边框、状态、结构秩序，而不是装饰性图形。

## 2. 核心气质

以后新项目如果要延续这套风格，AI 必须优先满足下面这些关键词：

- 技术工具感高于品牌宣传感。
- 紧凑密度高于宽松留白。
- 信息扫描效率高于视觉表演。
- 低饱和中性色底，高识别度单一强调色。
- 交互反馈克制，避免花哨动画。
- 文案直接、工程化、少废话。

一句更直接的话：

“做成一个让人想立刻开始操作的专业工具台，而不是一个解释自己多厉害的宣传站。”

## 3. 页面层级模型

### 3.1 首页模型

入口页不是传统 SaaS landing page，而是“工具集合入口页”。

应包含以下特征：

- 有明显的主视觉区，但不是巨大的 hero banner。
- hero 区更像技术控制台封面：网格、微弱光晕、渐变、边框层次。
- 首屏马上给出主要入口，不让用户滚很久才看到可操作内容。
- 功能卡片直接列出，强调“进入使用”。
- 入口页文案是说明当前能力范围，不讲空泛愿景。

入口页应给人的感觉：

- “这里有一组可直接使用的工具，我可以马上进。”
- 不是“欢迎来到一个未来平台，请先阅读我们是谁”。

### 3.2 工具页模型

工具页必须默认采用“工作台”结构，而不是说明文档结构。

推荐自上而下的骨架：

```text
页面
├─ 顶部工具栏 toolbar
├─ 主工作区 workspace
│  ├─ 左侧输入 / 源数据
│  └─ 右侧输出 / 结果
└─ 底部辅助区
   ├─ 选项 / 参数
   └─ 摘要 / 日志 / 结果信息
```

对 code-heavy 页面，优先使用这个具体版式：

```text
<main>
  <toolbar />
  <editors-grid columns=2 />
  <bottom-row left=options right=summary_or_log />
</main>
```

关键约束：

- 工具栏放在最上面，操作必须前置。
- 输入和输出尽量等宽双栏，桌面端强调对照效率。
- 日志、摘要、选项放在次级区域，不抢主操作区。
- 不要在工具页顶部塞一整屏“使用说明”。

## 4. 视觉系统规则

### 4.1 配色策略

当前这套风格的配色逻辑是：

- 默认深色模式。
- 以接近 zinc/neutral 的深灰黑做底。
- 用绿色系作为主强调色。
- 浅色模式保持同一套 token 语义，只替换具体值。

复用时要遵循：

- 基础背景用低饱和中性色，不要纯黑纯白硬切。
- 主强调色只保留一个核心方向，当前推荐继续用绿色系。
- 成功、警告、危险、信息色只用于状态，不要滥用。
- 所有颜色优先走 token，不要在组件里散落十六进制。

风格上的关键不是“绿色”本身，而是：

- 中性底色足够稳。
- 强调色足够聚焦。
- 状态颜色可读且不过度兴奋。

### 4.2 面板风格

这套风格中的大多数容器都遵守同一个公式：

- 1px 细边框。
- 中小圆角，常用 `lg` / `xl`。
- 半透明或轻混合背景。
- 轻阴影，强调层次但不做浮夸卡片。

推荐面板气质：

- 看起来像“仪表面板”或“工具舱段”。
- 不是玻璃拟态炫技，也不是完全扁平。

### 4.3 字体与排版

这套风格的字体逻辑：

- 正文字体：简洁、现代、可长时间阅读。
- 标题字体：与正文统一，不做强烈品牌化切换。
- 代码字体：清晰、窄一些、工程感强。

排版规则：

- 默认字号以 `12px` 到 `16px` 为主。
- 标题强调层级，不靠超大字号制造气势。
- 表单标签、元信息、状态文本通常偏小，但必须清楚。
- 字间距、字重变化很克制。

如果是工具页：

- 主要标题通常不需要超过 `sm` 到 `lg` 级别。
- 面板标题更像 section label，而不是营销标题。

## 5. 布局密度规则

这是一个偏高密度系统，AI 在生成页面时必须默认：

- 优先紧凑，不要先做宽松大留白。
- 按钮、筛选器、标签、元信息允许靠近，但不能拥挤到难以点击。
- 卡片 padding 中等偏小。
- 区块间距明确，但不拉太开。

简单判断标准：

- 如果页面第一屏只能看到标题和几句说明，说明太空了。
- 如果用户第一屏能同时看到操作区、状态、部分结果，密度通常才是对的。

## 6. 交互与动效规则

这套风格的交互不是靠强动画驱动，而是靠“稳定反馈”。

必须遵守：

- hover 主要改边框、背景、文字色。
- 少量 `translateY(-1px)` 可以用于入口卡片，但只做轻微提气。
- 不做大面积弹跳、发光扫过、长时过渡。
- 不要给全局背景色、文字色切换加明显 transition，否则明暗切换会闪。
- 焦点态必须清晰，通常用 accent 色描边 + 轻外发光。

交互的目标是：

- 用户知道自己点到了什么。
- 用户知道当前系统处于什么状态。
- 页面不要“很会动”，而要“很会工作”。

## 7. 图标使用规则

图标语言应保持统一，未来项目建议保持：

- 使用同一图标家族。
- 图标优先服务动作和状态识别。
- 不要给每一段文案都配图标。

建议规则：

- 按钮可以带图标。
- 状态 badge 可以带图标。
- 区块标题在有助于扫描时可以带图标。
- 长表单的每个 label 不要都加图标。
- 同一区域不要混搭多套图标风格。

## 8. 文案风格

文案应该是这个风格：

- 工程化。
- 直接。
- 少形容词。
- 少品牌腔。
- 少夸张承诺。

AI 在生成文案时要遵守：

- 用“处理结果”“运行概览”“当前范围”“已就绪”这种语言。
- 避免“革命性”“下一代”“极致体验”“强大平台”这类空话。
- 首页写能力边界，工具页写可执行动作。
- 优先告诉用户现在能做什么，而不是系统未来想做什么。

如果是提示文案：

- 短。
- 能落地。
- 尽量和当前状态强绑定。

## 9. 组件级约束

### 9.1 Header

header 应具备：

- sticky 顶部吸附。
- 半透明背景 + blur。
- 明确的底边线。
- 左侧品牌，中间导航，右侧工具区。
- 工具区可容纳搜索、主题切换、强调色切换等辅助控件。

header 的角色是：

- 稳定的产品壳层。
- 永远不喧宾夺主。

### 9.2 Toolbar

工具页 toolbar 是工作台核心，不是附属条。

应该遵守：

- sticky 在 header 下方。
- 左边放品牌 + 状态。
- 右边放操作按钮组。
- 按功能分组，并用 1px divider 断开。
- 主动作必须明显高于次动作。

操作分组推荐顺序：

1. 主动作 / 停止。
2. 输入侧操作。
3. 输出侧操作。
4. 重置或危险操作。

### 9.3 编辑器 / 输入输出区

如果页面是代码、文本、规则处理类工具：

- 默认双栏。
- 输入区和输出区尽量对称。
- 每个面板顶部有标题和元信息。
- 元信息用于字符数、行数、状态、耗时。
- 内联提示或错误信息贴着当前面板，不要飘去别处。

如果使用代码编辑器：

- 编辑器本身已经有边框时，外层面板和编辑器之间必须有一层 padding 缓冲。
- 不要让编辑器边框直接贴住父面板边框。

### 9.4 表单控件

控件风格应统一为：

- 中低高度。
- 细边框。
- 圆角偏大，但不是药丸级夸张。
- hover / focus 用 accent 边框强调。
- 说明文字放在 label 下方，字号更小、颜色更弱。

优先用共享组件，不要每个页面临时造一套 select / checkbox / button。

### 9.5 状态与反馈

状态呈现要“靠近发生地”：

- 运行状态放 toolbar badge。
- 输出错误放结果面板上方。
- 执行日志放底部单独面板。
- 复制成功、读取失败可用短 toast。

不要把所有反馈都堆到一个全局弹窗里。

## 10. 入口页特有的视觉语言

如果未来新项目也需要这类风格的入口页，建议复用下面这些元素：

- 技术感网格背景。
- 微弱 radial glow，不要大面积绚烂渐变。
- 英文/中文混合但克制，不搞大段 slogan。
- 工具卡片有轻微发光或 accent 渐变痕迹。
- “快速访问 / 状态台 / 当前范围”这类控制台式模块。

入口页需要的是“技术入口氛围”，不是“商业品牌发布会氛围”。

## 11. 响应式规则

移动端不是简单缩小，而是保持工作台逻辑成立。

必须遵守：

- toolbar 外层必须允许换行。
- toolbar actions 也必须允许换行。
- 不要给 actions 容器加 `flex-shrink: 0` 导致横向溢出。
- 双栏在中小屏要及时折成单栏。
- grid item 需要时显式加 `min-width: 0`，防止内容把父级撑爆。
- 目标是 375px 到 390px 宽度下不出现横向滚动。

移动端优先级：

- 先保证可操作。
- 再保证可扫描。
- 最后才是保持桌面端的视觉完整度。

## 12. 明确的 Do / Don't

### Do

- 做成技术工作台。
- 把操作入口前置。
- 用边框、面板、微渐变建立层次。
- 用 token 驱动颜色、间距、圆角和阴影。
- 用短文案解释当前能力和状态。
- 用统一图标家族做动作识别。
- 保持深浅模式语义一致。

### Don’t

- 不要做成营销 landing page。
- 不要上来就是夸张 hero、巨大标题、大片空白。
- 不要使用漂浮感很重的大阴影和复杂动画。
- 不要把帮助文档塞到工具页首屏前面。
- 不要让按钮组、筛选器、日志区互相抢主次。
- 不要在局部组件里乱写硬编码颜色。
- 不要混入多套不一致的按钮 / 输入框风格。

## 13. 给 AI 的直接执行指令

下面这段文字可以直接复制给 AI，作为新项目 UI 生成指令：

```md
请按当前这套 UI / UX 风格来设计，不要做成营销官网，要做成“工程师工作台 / 技术工具界面”。

必须满足：
- 整体风格偏技术感、冷静、紧凑、高密度、可扫描。
- 入口页像工具入口页，首屏直接给主要入口，不要大段品牌叙事。
- 工具页像工作台：顶部 sticky toolbar，主体是输入 / 输出双栏，底部是选项 + 日志/摘要。
- 默认使用低饱和中性色背景 + 单一绿色强调色，深色为主，同时支持浅色模式。
- 面板统一为细边框、中等圆角、轻阴影、轻微透明混合背景。
- 文案必须直接、工程化、少营销词，优先描述当前能做什么、当前状态是什么。
- 图标统一，主要用于动作按钮、状态 badge、局部语义锚点，不要图标泛滥。
- 动效克制，只允许轻微 hover、边框变化、少量 1px 浮动，不要炫技动画。
- 表单和工具条尽量紧凑，用共享组件思路统一按钮、选择器、复选框、编辑器样式。
- 响应式必须保证 375px~390px 不横向溢出，toolbar 和 actions 都能换行。

严格避免：
- 巨大 hero banner
- 大段宣传文案
- 过度留白
- 默认 admin 模板感
- 五颜六色的强调色并存
- 夸张阴影、霓虹、悬浮动画
- 把“说明文档”放在主工作区前面
```

## 14. 如果要继续贴近这套风格，可直接对齐这些实现特征

未来新项目如果想继续沿用这套风格，可以继续复用这些明确偏好：

- 顶部是 sticky shell header。
- 导航、主题切换、搜索入口使用胶囊或轻圆角控件。
- 工具页状态用 badge 表达“已就绪 / 处理中 / 失败”。
- 卡片 hover 只做轻微边框增强和极小位移。
- 元信息常用“字符数 / 行数 / 耗时 / 当前范围 / 运行概览”。
- 入口页模块化表达常用“当前能力、快速访问、运行概览”。
- 代码或文本工具优先使用真正的编辑器，不用原始 textarea 顶替。

## 15. 落地时的检查清单

AI 或人工在完成页面后，至少自查下面这些问题：

- 页面第一屏是否已经出现关键操作，而不是只有介绍？
- 工具页是否明显是“工作台”，而不是“说明页”？
- 信息密度是否足够高，但仍然可读？
- 主强调色是否只有一个清晰方向？
- 所有面板、按钮、输入框是否属于同一视觉语言？
- hover / focus / disabled / running 状态是否明确？
- 深浅主题切换是否语义一致、没有闪烁感？
- 移动端是否无横向滚动？
- 文案是否直接、专业、克制，没有营销腔？

如果上面大部分答案是“是”，那基本就对齐了这套 UI / UX 风格方向。
